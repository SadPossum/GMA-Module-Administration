# Administration v0.1 RBAC Migration

Administration v0.1 stored roles, permissions, principals, and assignments in the `admin` schema. Current GMA keeps those historical tables for recovery, but all active persisted authorization belongs to `Gma.Modules.AccessControl` in the `access` schema.

There is intentionally no automatic cross-module migration. A generic migration cannot safely decide whether a historical principal is an admin actor, user, service, or product-specific identity, nor can it resolve role-name conflicts or product permission changes. Small installations should recreate authority deliberately through AccessControl administration. Larger installations should use a reviewed, one-off product tool or a future explicit GMA Extensions importer built against module contracts.

## Before Cutover

1. Schedule a maintenance window that prevents role and assignment mutations.
2. Back up the database and prove that the backup can restore both the `admin` and `access` schemas.
3. Record counts and exports for `admin.principals`, `admin.roles`, `admin.role_permissions`, and `admin.principal_roles`.
4. Inventory every historical role name, permission code, principal id, and assignment scope. Resolve invalid or obsolete permission codes before import.
5. Decide the AccessControl subject kind for every principal. Historical Administration actors normally map to `admin-actor`, but a product must confirm that identity rather than infer it from the identifier alone.
6. Confirm the composing host includes current AccessControl migrations, Administration audit persistence, and the Administration-to-AccessControl authorization bridge.

## Mapping

Use the following mapping only after the product has validated the identity and scope semantics:

| Administration v0.1 | AccessControl |
| --- | --- |
| `principals.Id` | Explicit `AccessSubject` kind plus the same stable subject id |
| `roles.Name` | Global compatibility-role name |
| `role_permissions.PermissionCode` | Exact role permission grant |
| empty `principal_roles.TenantId` | `global` assignment scope |
| non-empty `principal_roles.TenantId` | normalized `tenant:<tenant-id>` assignment scope |

Role definitions are global compatibility roles in both models. Assignment scope is separate from the role definition. Do not turn a tenant id into part of a role name, and do not grant a concrete permission at `global` expecting it to inherit into tenant scopes unless that permission's registered descriptor explicitly allows the required scope policy.

## Supported Cutover

1. Apply AccessControl migrations without dropping or modifying the legacy Administration tables.
2. Bootstrap one recovery owner through `admin bootstrap --actor <id> --yes` only when AccessControl has no assignments.
3. Recreate each approved role with `admin roles create` and its exact permission set with `admin roles grant`.
4. Recreate each approved assignment with an explicit target kind, target id, role, and `global` or normalized tenant scope.
5. For a larger migration, implement the same sequence through `IAccessControlRoleProvisioner` or another reviewed AccessControl Contracts surface in a product-owned one-off tool. Do not read or write another module's EF context or tables from Administration.
6. Keep the maintenance window active until verification succeeds.

## Verification

- Compare role, permission, and assignment inventories with the approved source export, accounting for deliberately removed obsolete authority.
- Evaluate representative allowed and denied decisions for every role at global, tenant, and descendant scopes used by the product.
- Verify every operator identity uses the intended AccessControl subject kind.
- Verify at least one recovery owner exists and that final-owner protection is active.
- Exercise one authorized and one denied admin operation and confirm both produce Administration audit records.
- Save the migration manifest, comparison output, test evidence, application commit, and database backup identifier together.

## Rollback

If verification fails, keep admin mutations stopped. Restore the pre-cutover backup or remove only the imported AccessControl assignments and roles using the reviewed migration manifest. Revert the composing application to the previously verified version if needed. The untouched Administration v0.1 tables remain the recovery source.

Do not drop the legacy tables during cutover. Removal requires a separate product migration after the new authorization model has operated successfully, backup retention is satisfied, and rollback no longer depends on the old records.
